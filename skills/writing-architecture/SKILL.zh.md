# 编写 ARCHITECTURE.md

## 概述

ARCHITECTURE.md 是一份稳定的项目知识文档。它帮助新人建立心智模型，帮助贡献者回忆设计意图，帮助 AI agent 理解项目约束。它回答"为什么要这样设计？"——而不是"代码在哪里？"

**核心原则：** 写下你第一天加入项目时希望知道的东西——那些需要几个月才能自然学到的、一旦理解就很少改变的知识。

## 适用场景

- 为新项目或现有项目创建 ARCHITECTURE.md
- 修订现有的 ARCHITECTURE.md（每年 revisiting 几次，不是每次提交都改）
- 跳过：项目不到 ~1k 行代码、CLAUDE.md 已经充分覆盖架构

## 结构

每份 ARCHITECTURE.md 应按顺序包含以下章节：

### 1. 项目定位

3-8 行描述项目是什么、解决什么问题、为谁服务。读者读完这一节就能理解项目的目的。

**稳定性判断：** 只有项目的目标用户或核心问题发生变化时才需要修改。

### 2. 设计哲学

3-7 条设计原则。每条包含：名称、一句话阐述、选择背后的理由。重点关注"为什么选 A 不选 B"的决策，不是模块级的实现细节。

**稳定性判断：** 只有项目发生根本性方向转变时才需要修改。

### 3. 核心概念

5-10 个读者必须理解的项目特有概念。每个概念包含：名称、一句话定义、与其他概念的关系。只收录"不理解就无法理解项目"的概念。

**稳定性判断：** 概念构成项目的抽象骨架。实现可以变，概念不变。

### 4. 系统运转

2-4 个端到端场景展示系统如何工作。每个场景：什么触发了它、经过了哪些核心概念、产生了什么产出。重点关注流转路径，不深入每个步骤的内部实现。

**稳定性判断：** 只有系统的根本运转模式发生变化时才需要修改。

### 5. 模块职责

对每个顶层目录或主要模块，2-4 行描述：设计意图（为什么要有这个模块）和职责边界（它负责什么、不负责什么）。按名称提及关键文件或类型供 symbol search，不枚举内部文件。

**稳定性判断：** 模块职责在项目成熟后很少变化。

### 6. 架构约束

- **层边界：** 对每个边界，说明每一边不知道另一边的什么
- **不变量：** 跨模块的系统级规则，特别是"缺席型"不变量（"X 从不依赖 Y"）
- **设计约束：** 有意为之的限制（"技能永远是纯文本，不包含可执行代码"）

即使是单体项目也有概念上的层（入口 → 业务逻辑 → 数据）。描述那些。如果项目确实没有分层，解释为什么这是有意为之。

**稳定性判断：** 架构约束往往在项目创建时就已确定。

## 规则

### 应该做

- **命名重要的文件、模块和类型。** 使用精确名称以便读者用 symbol search 查找。
- **标出架构不变量。** 特别是代码中"缺席"的东西——"X 层不依赖 Y 层"在读 X 层代码时是不可见的。
- **显式描述边界。** 说明每层不知道其邻居的什么。
- **保持简短。** 目标 80-120 行。每个反复贡献者都要读它。
- **只包含稳定的事实。** 避免版本号、具体 commit hash 或每次发布都会变的细节。
- **先回答"为什么"再回答"是什么"。** 设计意图比结构描述更有价值。

### 不应该做

- **不要链接文件。** 链接会过期。使用文件名，让读者自己搜索。
- **不要枚举内部文件。** "包含辅助 prompt"就够了——不要列出每个 prompt 文件名。
- **不要包含特定版本的值。** "版本遵循 X 模式"是稳定的；"当前版本是 5.0.7"不是。
- **不要解释模块内部如何运作。** 那是内联文档或独立文档的事。这份文档说的是东西"是什么"和"为什么存在"，不是"怎么运作"。
- **不要写第二个 README。** 这是给贡献者的，不是给用户的。聚焦于项目知识，不是功能描述。
- **不要包含测试/构建约定。** 那些属于 CLAUDE.md 或 CONTRIBUTING.md，不属于稳定的架构知识。

## 常见错误

| 错误 | 为什么有问题 | 修正方式 |
|------|-------------|---------|
| 跳过设计哲学 | 新人错过了决策背后的"为什么" | 始终包含 3-7 条带理由的原则 |
| 核心概念写成词典样式 | 只有定义没有关系，无法建立心智模型 | 包含概念之间的关系 |
| 系统运转解释了内部实现 | 应该展示流转路径，不是实现细节 | 描述触发 → 流转 → 产出，不是每步怎么做 |
| 模块职责变成文件列表 | 退化为旧的代码地图行为 | 聚焦设计意图，只命名 2-3 个关键项 |
| 包含版本号或 hash | 每次发布都会变，文档变错 | 描述模式或约定 |
| 面向用户而非贡献者 | 受众错误——这是项目知识 | 聚焦设计意图，不是功能描述 |
| "我的项目没有架构约束" | 即使简单项目也有有意为之的限制 | 描述项目刻意不做的事 |

## 质量检查清单

完成前，验证：

- [ ] 项目定位：新人 30 秒内知道"这是什么"？
- [ ] 设计哲学：每条原则都解释了"为什么选这个方向"？
- [ ] 核心概念：覆盖了"不理解就无法理解项目"的所有概念？
- [ ] 系统运转：至少有一个端到端场景？
- [ ] 模块职责：每个模块说了"为什么存在"和"职责边界"？
- [ ] 架构约束：缺席型不变量是否显式声明？
- [ ] 稳定性：没有版本号、hash 或文件枚举？
- [ ] 长度：120 行以内？
- [ ] 没有链接，只用名称供 symbol search？
