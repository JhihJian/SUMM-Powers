# Feedback Loop Construction Methods

## Overview

Debugging is building a feedback loop that tells you what's wrong. The right loop makes the bug obvious; the wrong loop wastes time.

**Core principle:** Build the cheapest feedback loop that reveals the bug. Escalate only when it doesn't.

## The 10 Methods (Priority Order)

### 1. Failing Test

The fastest loop. Write a test that reproduces the bug.

```
Red → observe failure → fix → green → done
```

**When to use:** Bug is deterministic and you can isolate it in a test.

**Example:**
```typescript
test('cart applies 20% discount for 5+ items', () => {
  const cart = new Cart();
  for (let i = 0; i < 5; i++) cart.add(item);
  expect(cart.total()).toBe(expected - expected * 0.2);
  // Fails: discount not applied for exactly 5 items
});
```

### 2. curl / HTTP Script

Hit the API directly. Skip the UI, skip the framework.

```bash
curl -v -X POST http://localhost:3000/api/cart \
  -H 'Content-Type: application/json' \
  -d '{"items": [{"id": 1, "qty": 5}]}'
```

**When to use:** Bug is in an HTTP endpoint or service boundary.

### 3. CLI Invocation

Call the function/module directly from command line.

```bash
node -e "const { parse } = require('./parser'); console.log(parse(input))"
```

**When to use:** Bug is in a library function, parser, or pure logic.

### 4. Headless Browser

Automate browser interaction without manual clicking.

```typescript
import { chromium } from 'playwright';
const browser = await chromium.launch();
const page = await browser.newPage();
await page.goto('http://localhost:3000');
await page.fill('#email', 'test@example.com');
await page.click('#submit');
console.log(await page.textContent('.result'));
```

**When to use:** Bug only manifests in browser context (DOM, network, rendering).

### 5. Replay Captured Trace

Capture a real session, replay it to reproduce.

```bash
# Record
tcpdump -i lo -w trace.pcap port 3000

# Replay or analyze
tshark -r trace.pcap -Y "http.request"
```

**When to use:** Bug involves network protocol, message ordering, or timing.

### 6. Disposable Harness

A one-off script that exercises the buggy code path with minimal setup.

```python
# harness.py — delete after bug is found
from module import buggy_function
result = buggy_function(edge_case_input)
print(f"Got: {result}, Expected: {expected}")
```

**When to use:** Bug requires specific state or data that's hard to set up in tests.

### 7. Property-Based / Fuzz Loop

Generate random inputs to find the triggering condition.

```python
from hypothesis import given, strategies as st

@given(st.integers(), st.integers())
def test_division_commutative_check(a, b):
    # Will shrink to minimal failing case
    assert buggy_function(a, b) == expected(a, b)
```

**When to use:** Bug is edge-case dependent but you don't know which edge case.

### 8. Bisection Harness

Binary search through code history or input space.

```bash
# Git bisect for code regression
git bisect start
git bisect bad HEAD
git bisect good v1.2.0
# Git checks out middle commit; you run test, mark good/bad

# Input bisect — halve input until bug appears
```

**When to use:** Bug is a regression (worked before, broken now) or triggered by specific input ranges.

### 9. Differential Loop

Run two versions side by side and compare outputs.

```bash
# Old version
old_version < input > old_output
# New version
new_version < input > new_output
# Compare
diff old_output new_output
```

**When to use:** Bug is a behavioral change — output differs from expected but not obviously wrong.

### 10. Human-in-the-Loop (HITL) Bash Script

A script that pauses for human judgment at decision points.

```bash
#!/usr/bin/env bash
for file in可疑文件列表; do
  echo "=== $file ==="
  head -5 "$file"
  read -p "Investigate? (y/n) " answer
  [[ "$answer" == y ]] && vim "$file"
done
```

**When to use:** Bug requires human judgment (visual verification, domain knowledge) that can't be automated.

## Escalation Rule

```
If current method doesn't reveal bug in 10 minutes → escalate to next method.
Don't keep thrashing with a loop that isn't working.
```

## Anti-Pattern: Building the Wrong Loop

| Loop Type | Wrong When | Use Instead |
|-----------|-----------|-------------|
| Manual browser clicking | Bug is in API logic | curl script (#2) |
| Adding print statements | Bug is non-deterministic | Fuzz loop (#7) or bisection (#8) |
| Reading code staring | Bug is state-dependent | Disposable harness (#6) |
| Re-running test hoping | Bug is timing-related | Replay trace (#5) or condition-based waiting |
```

# Performance Regression Debugging

## Overview

Performance bugs are different from correctness bugs. Logs and intuition are unreliable. You need measurements first.

**Core principle:** Measure before you bisect. Logs lie, benchmarks don't.

## The Wrong Way: Log-Driven Performance Debugging

```
"Feels slow" → add timing logs → see log says X is slow → optimize X → still slow
```

This fails because:
- Logs measure wall clock time, which includes unrelated waits
- The slowest log line isn't always the bottleneck
- Logs don't show cumulative effects (N small delays > 1 big delay)

## The Right Way: Baseline-First Debugging

### Step 1: Establish a Baseline

Before anything else, measure the current state objectively.

```bash
# Benchmark with realistic data
time your-command --input=test-data.json

# Or with a proper benchmark harness
benchmark --iterations=100 your-function
```

Record the number. This is your baseline.

### Step 2: Identify Regression Window

If performance was good before:

```bash
# Git bisect for the regression commit
git bisect start
git bisect bad HEAD          # Current: slow
git bisect good v2.0.0      # Known: fast
# Git checks out commits; you run benchmark at each step
```

### Step 3: Profile, Don't Guess

Use a profiler to find where time is actually spent:

```bash
# Node.js
node --prof your-app.js
node --prof-process isolate-*.log

# Python
python -m cProfile -s cumulative your_script.py

# Go
go test -cpuprofile=cpu.out && go tool pprof cpu.out
```

### Step 4: Hypothesis with Measurements

Form hypothesis: "I think X is slow because Y."

Test with targeted measurement:
```bash
# Before optimization
benchmark-x --baseline=before.json

# After optimization
benchmark-x --baseline=after.json

# Compare
compare-benchmarks before.json after.json
```

### Step 5: Verify Regression is Fixed

Run the same benchmark from Step 1. Confirm improvement.

```
Before: 2.3s average
After:  0.4s average
Improvement: 5.75x ✓
```

## Common Performance Anti-Patterns

| Anti-Pattern | Why It Fails | Fix |
|-------------|-------------|-----|
| "Add timing logs everywhere" | Logs have overhead; wall clock ≠ CPU time | Profile first, log second |
| "It's probably the database" | Assumption without measurement | Profile to find actual bottleneck |
| "Optimize the hot loop" | Hot loop may not be the bottleneck | Profile to confirm before optimizing |
| "Rewrite in faster language" | May not address algorithmic issue | Measure first, optimize algorithm second |
| "Cache everything" | Caching wrong thing adds overhead | Measure hit rates and latency impact |

## Non-Deterministic Performance Bugs

Some performance issues only appear under specific conditions (load, data size, cache state).

**Strategy:** Increase reproduction rate, don't chase 100% reproduction.

1. **Stress test:** Run under higher load to amplify the issue
2. **Data scaling:** Test with 10x, 100x data to find threshold
3. **Cache clearing:** Disable caches to see if bug is cache-related
4. **Isolation:** Run on dedicated machine to eliminate noise

**Goal:** Make the bug reproducible enough to profile, not 100% reproducible.
```

## Tagged Debug Logs

When adding debug logging to trace a bug:

1. **Tag every log line** with a unique prefix: `[DEBUG-a4f2]`
2. **Log before the operation**, not after it fails
3. **Include context:** relevant variables, state, environment

```typescript
console.error('[DEBUG-a4f2] Before git init:', { directory, cwd: process.cwd() });
```

4. **Clean up with one command:**

```bash
git grep -l 'DEBUG-a4f2' | xargs sed -i '/DEBUG-a4f2/d'
```

**One tag. One grep to clean. No leftover debug noise.**

## Non-Deterministic Bugs

When a bug doesn't reproduce every time:

- **Goal is not 100% reproduction.** Goal is raising reproduction rate to a debuggable level.
- **Stress the system:** Run the operation 100 times, log each attempt.
- **Freeze randomness:** Set fixed seeds, disable timeouts, remove concurrency.
- **Bisect conditions:** Remove variables one at a time until bug frequency changes.
- **If you can make it fail 1 in 10 times,** you can debug it. 1 in 1000 is hard. Improve the ratio.

See `performance-regression.md` for performance-specific non-deterministic debugging.

# Deep Modules

## Overview

A deep module has a small interface and a large implementation. It gives callers leverage (simple API, powerful behavior) and maintainers locality (changes stay inside).

**Core principle:** Test the interface, not the implementation. Deep modules make this natural — the interface is small and stable.

## Key Concepts

### Depth

Interface simplicity relative to implementation complexity.

```
Shallow: 5 public methods, each does almost nothing → callers do all the work
Deep:    1 public method, does everything internally → callers just call it
```

**Deep is good.** Callers get power from a simple call. Maintainers change internals freely.

### Leverage

What callers gain from depth. A deep module's interface is a handle on complex behavior.

```
High leverage: file.read() → handles buffering, encoding, OS calls, errors
Low leverage:  file.open(), file.seek(), file.readBytes(), file.close() → caller orchestrates
```

### Locality

What maintainers gain from depth. Changes to implementation don't affect callers.

```
High locality: Change file.read() buffering strategy → zero caller changes
Low locality:  Change readBytes() contract → every caller must update
```

### Seam

Where interface meets implementation. The point where behavior can change without modifying callers.

```
Seam example: Database interface
  callers → interface.save(data)
  today  → PostgreSQL implementation
  tomorrow → Redis implementation
  Callers unchanged.
```

### Deletion Test

Imagine deleting the module. Where does the complexity go?

```
If complexity vanishes → module was a pass-through (shallow)
If complexity scatters to N callers → module was earning its depth
```

## Testing Deep vs Shallow Modules

### Deep Module Testing (Easy)

```typescript
// Interface is small — test it completely
test('file.read returns file contents', () => {
  fs.writeFileSync('/tmp/test.txt', 'hello');
  expect(readFile('/tmp/test.txt')).toBe('hello');
});

// Internal buffering strategy? Doesn't matter. Interface works.
```

### Shallow Module Testing (Hard)

```typescript
// Interface is large — every method needs tests
test('open returns handle', ...);
test('seek moves position', ...);
test('readBytes returns N bytes', ...);
test('close releases handle', ...);
test('forgetting close leaks', ...);
test('seek then read returns right bytes', ...);
// Combinatorial explosion of method interactions
```

## Deepening Shallow Modules

When a module feels shallow:

1. **Ask:** What is the caller trying to accomplish? (Not: what operations does it need?)
2. **Combine:** Can two methods become one with a sensible default?
3. **Hide:** Can internal details become truly internal (not exposed as public methods)?
4. **Rename:** Does the interface describe WHAT it does or HOW it does it?

**Warning:** Deepening requires understanding all callers. Don't deepen in isolation.

## Anti-Patterns

| Anti-Pattern | Signal | Fix |
|-------------|--------|-----|
| God object | One class with 50 methods | Split by responsibility |
| Pass-through | Module just delegates | Remove it or make it deep |
| Feature envy | Method uses another object's data more than its own | Move the method |
| Shallow wrapper | Adds no behavior, just renames | Remove or deepen |

## Relation to TDD

TDD naturally produces deep modules because:
- You write the interface you wish existed (the test's API)
- Then implement the simplest thing that satisfies it
- The test forces a small, caller-centric interface

If tests are hard to write, the module is too shallow. Listen to the test pain.
```

# Interface Design for Testability

## Overview

If code is hard to test, it's hard to use. Test difficulty is a design signal, not a testing problem.

**Core principle:** Design interfaces that are easy to call from both production code and tests.

## The Testability Checklist

When designing an interface, ask:

1. **Can I create the object with test data?** If construction requires 5 real services, the interface is too coupled.
2. **Can I observe the result?** If the only way to verify behavior is checking a database, the return value is wrong.
3. **Can I control time and randomness?** If the code uses `Date.now()` or `Math.random()` internally, I can't test deterministically.
4. **Can I replace dependencies?** If the class creates its own database connection, I can't substitute a test one.

## Design Patterns for Testability

### Dependency Injection

Pass dependencies in, don't create them internally.

```typescript
// Hard to test: creates its own dependency
class UserService {
  private db = new Database(); // Can't substitute
  async getUser(id: string) { return this.db.query(...); }
}

// Testable: accepts dependency
class UserService {
  constructor(private db: Database) {}
  async getUser(id: string) { return this.db.query(...); }
}

// In test:
const mockDb = { query: vi.fn().mockResolvedValue({ name: 'Test' }) };
const service = new UserService(mockDb);
```

### Pure Functions

Same input → same output. No side effects. Trivially testable.

```typescript
// Pure: easy to test
function formatCurrency(amount: number, currency: string): string {
  return `${currency} ${amount.toFixed(2)}`;
}

// Impure: hard to test
function displayTotal(): void {
  const cart = getGlobalCart();       // Depends on global state
  const rate = fetchExchangeRate();   // Side effect
  document.getElementById('total').textContent = ...; // Side effect
}
```

### Return Values Over Side Effects

Functions that return data are easier to test than functions that perform actions.

```typescript
// Returns data: testable
function computeDiscount(cart: Cart): number {
  return cart.items.length >= 5 ? 0.2 : 0;
}

// Performs action: hard to test
function applyDiscount(cart: Cart): void {
  cart.total *= cart.items.length >= 5 ? 0.8 : 1; // Mutates
  updateUI(cart.total);                             // Side effect
}
```

### Configurable Time Sources

```typescript
// Hard to test: uses real time
function isExpired(token: Token): boolean {
  return Date.now() > token.expiresAt; // Non-deterministic
}

// Testable: accepts time source
function isExpired(token: Token, now: number = Date.now()): boolean {
  return now > token.expiresAt;
}

// In test:
expect(isExpired({ expiresAt: 1000 }, 999)).toBe(false);
expect(isExpired({ expiresAt: 1000 }, 1001)).toBe(true);
```

## Anti-Patterns

| Anti-Pattern | Why Untestable | Fix |
|-------------|---------------|-----|
| Singleton/global state | Can't reset between tests | Dependency injection |
| `new` inside methods | Can't substitute | Accept dependency in constructor |
| Void methods with side effects | Can't observe result | Return the result |
| Hidden time dependence | Non-deterministic | Accept time source as parameter |
| God constructor (10 params) | Hard to construct for tests | Group into config objects; reduce coupling |

## The Design Feedback Loop

```
Write test → Hard to set up? → Interface is too coupled
           → Hard to assert? → Return value doesn't expose behavior
           → Test is fragile? → Testing implementation, not interface
           → Test is simple? → Good design ✓
```

**Listen to test pain.** It tells you about design problems, not testing problems.
```

# Mocking Guide

## Overview

Mocks isolate the unit under test, but they also couple tests to implementation. Use them judiciously.

**Core principle:** Mock at system boundaries (external services, I/O), not at internal seams.

## The Decision Tree

```
Need to isolate a dependency?
├─ Is it external (API, database, filesystem)?
│  └─ YES → Mock it. Use interface-based mock or test double.
│
├─ Is it slow (network, sleep, heavy computation)?
│  └─ YES → Mock it, but mock at the lowest level (the slow operation, not the module using it).
│
├─ Is it non-deterministic (time, random, UUID)?
│  └─ YES → Inject a controllable source, don't mock the module.
│
├─ Is it an internal module in your codebase?
│  └─ Use the REAL implementation.
│     If real impl is hard to use in tests → the real impl has a design problem. Fix the design.
│
└─ Is it a module you're actively developing?
   └─ Use the REAL implementation. Mocking it hides integration bugs.
```

## When to Mock

### Mock: External Services

```typescript
// Mock the HTTP client, not the service wrapper
vi.mock('./http-client', () => ({
  post: vi.fn().mockResolvedValue({ id: '123' })
}));

// Use real service logic that calls the mocked client
const service = new PaymentService(httpClient);
```

### Mock: Slow Operations

```typescript
// Mock at the level of the slow thing
vi.mock('./email-sender', () => ({
  send: vi.fn().mockResolvedValue(true)
}));

// Don't mock NotificationService — test its logic with mocked email
```

### Mock: Non-Deterministic Sources

```typescript
// Don't mock modules that use time — inject a time source
class TokenValidator {
  constructor(private getTime: () => number = Date.now) {}

  isValid(token: Token): boolean {
    return token.expiresAt > this.getTime();
  }
}

// In test:
const validator = new TokenValidator(() => 1000);
```

## When NOT to Mock

### Don't Mock: Internal Modules

```typescript
// BAD: Mocking an internal calculator
vi.mock('./calculator');
test('cart totals items', () => {
  const cart = new Cart(mockCalculator);
  // Test proves nothing — you're testing that mock works
});

// GOOD: Use real calculator
test('cart totals items', () => {
  const cart = new Cart(new Calculator());
  cart.add(item100);
  cart.add(item200);
  expect(cart.total()).toBe(300);
});
```

### Don't Mock: The Module Under Test

```typescript
// BAD: Mocking what you're testing
vi.mock('./user-service');
test('user service creates user', () => {
  // What are you even testing?
});

// GOOD: Test real behavior
test('user service creates user', async () => {
  const service = new UserService(testDb);
  const user = await service.create({ name: 'Alice' });
  expect(user.name).toBe('Alice');
  expect(await testDb.find(user.id)).toBeDefined();
});
```

### Don't Mock: Data Structures

```typescript
// BAD: Mocking a data object
const mockUser = { getName: vi.fn(), getEmail: vi.fn() };

// GOOD: Use a real data object
const user = { name: 'Alice', email: 'alice@example.com' };
```

## Mocking Levels (Best to Worst)

| Level | Approach | Coupling | Best For |
|-------|---------|---------|----------|
| 1. Interface | Implement test double for interface | Low | External services |
| 2. Function injection | Pass function as parameter | Low | Strategies, policies |
| 3. Module mock | `vi.mock('module')` | Medium | Slow/unavailable deps |
| 4. Method stub | `obj.method = vi.fn()` | High | Last resort |

**Prefer lower coupling.** If you find yourself at level 4, consider a design change.

## Mock Maintenance Cost

Every mock is a contract between test and implementation. When implementation changes:

| Mock Type | Breaks When | Fix Cost |
|-----------|------------|---------|
| Interface mock | Interface changes (rare) | Update test double |
| Function injection | Function signature changes | Update call sites |
| Module mock | Any internal change | Rewrite mock setup |
| Method stub | Any method change | Update every stub |

**More mocks = more brittle tests.** Prefer fewer mocks, more real code.

## Quick Reference

| Situation | Action |
|-----------|--------|
| External API | Mock at HTTP boundary |
| Database | Use in-memory test DB if possible; mock only if setup is prohibitive |
| Filesystem | Use temp directories; mock only for error simulation |
| Time/random | Inject controllable source |
| Internal module | Use real implementation |
| Module under test | Never mock |
| "I need to mock X" | Ask: why can't I use the real X? |
```

## Vertical Slices vs Horizontal Slices

### Anti-Pattern: Horizontal Slicing

Writing all tests first, then all implementation:

```
RED: test1, test2, test3, ..., testN → all fail
GREEN: impl1, impl2, impl3, ..., implN → some pass
```

This fails because:
- Tests written without implementation feedback miss edge cases
- Implementation drifts from test assumptions
- No working software until the end
- Can't verify tests fail for the right reasons (you wrote them all at once)

### Correct: Vertical Slicing (Tracer Bullets)

One test → one implementation → repeat:

```
RED: test1 → GREEN: impl1 → REFACTOR
RED: test2 → GREEN: impl2 → REFACTOR
...
RED: testN → GREEN: implN → REFACTOR
```

Each cycle produces working, tested software. Each test benefits from implementation knowledge of the previous cycle.

**The Red-Green-Refactor cycle IS vertical slicing.** Don't batch the cycles.
```
