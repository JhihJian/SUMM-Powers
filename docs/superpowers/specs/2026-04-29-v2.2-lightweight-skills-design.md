> 范围: A3 (caveman) + A6 (zoom-out) + A7 (grill-me)

## 概述

v2.2 新增三个轻量技能，聚焦"减少噪音、提升决策质量"。不涉及外部依赖或架构变更。

## A3: caveman — 替换 `less` 技能

**技能目录**: `skills/less/`（保持不变，替换 SKILL.md 内容）

### 核心能力

对话级 token 优化。将 verbose 输出压缩为 caveman 格式，约节省 75% token。

### caveman 规则

- 格式: `[thing] [action] [reason]. [next step].`
- 删掉冠词、填充词、礼貌用语
- 保留技术术语、文件路径、命令
- 一行一个事实，不做段落展开

### 触发与退出

- **激活**: 用户说 "use caveman"、"简洁模式"、"少说话多做事"，或调用 `/less`
- **退出**: 用户说 "normal mode"、"正常模式"、"详细说"
- 不自动激活，必须用户显式请求
- 一旦激活，当前会话所有后续输出使用 caveman 格式，直到退出

### 不适用

- 用户明确要求详细解释时
- 展示设计/spec 时（这些有专用格式）
- 错误信息需要完整上下文时
- brainstorming 和 writing-plans 流程中（这些有固定输出格式）

### 示例

```
# 正常模式:
好的，我来分析一下这个问题。根据你的描述，这个 bug 可能是因为在处理并发请求时，
锁的释放顺序不正确导致的。建议你先检查 Redis 连接池的配置，然后...

# Caveman 模式:
[concurrent lock release] [order incorrect] [race condition on Redis pool].
[check] Redis connection pool config. [add] ordered lock release.
```

### 取舍

原有的 1-1-3-1 审阅格式将被移除。该格式使用场景窄（仅文档审阅），caveman 覆盖更广（所有对话输出）。

## A6: zoom-out — 上下文提升

**技能目录**: `skills/zoom-out/`（新建）

### 核心能力

从当前工作细节中"退后一步"，展示宏观模块关系和当前位置。帮助用户避免"迷失在细节中"。

### 触发条件

- 用户说 "zoom out"、"退后一步"、"看看大局"、"整体架构"、"宏观视角"
- 用户迷迹在细节中、反复修改同一区域却没进展

### 输出格式（固定结构）

```
## 当前位置
[2-3 句话说明正在做什么，在整个项目中的位置]

## 模块关系
[简化关系图，最多 8 个节点，只展示直接相关模块]

## 建议下一步
[基于宏观视角的 1-2 个建议]
```

### 关键规则

- 关系图节点不超过 8 个，层级不超过 3 层
- 不重复用户已知的信息，只展示"盲区" — 用户可能忽略的依赖和影响
- 如果项目有 CONTEXT.md，引用其中的术语体系
- 纯只读，不修改任何文件

### 适用场景

- 开始新功能前，确认改动影响范围
- debug 陷入细节时，重新定位
- code review 时，理解 PR 在整体中的位置

### 不适用

- 已经清楚全貌的情况
- 纯前端/单文件改动（无需宏观视角）

## A7: grill-me — 轻量烤问

**技能目录**: `skills/grill-me/`（新建）

### 核心能力

快速决策验证 — 持续追问直到决策树的每个分支都被探索。不产出文档，对话即结果。

### 与 brainstorming 的区别

| 维度 | brainstorming | grill-me |
|------|---------------|----------|
| 流程 | 9 步 checklist + spec 输出 | 无固定流程，纯对话 |
| 输出 | 设计文档 + git commit | 无产出，对话即结果 |
| 终态 | invoke writing-plans | 用户说"够了"或问题穷尽 |
| 深度 | 完整设计（架构、数据流、错误处理） | 聚焦单一决策点 |
| 重量 | 适合新功能/大改动 | 适合单个决策/想法验证 |

### 追问策略（按优先级）

1. **假设挖掘** — "你假设了 X，如果 Y 呢？"
2. **边界条件** — "这个方案在 Z 情况下会怎样？"
3. **替代方案** — "为什么不选 W？"
4. **连锁效应** — "做了 X 之后，对 A/B/C 有什么影响？"
5. **收敛确认** — "所以你的核心决策是 P，条件是 Q，对吗？"

### 关键规则

- 不假设任何上下文，从用户当前的描述出发
- 每次只问一个问题
- 如果用户已有明确答案，不重复追问
- 发现用户想清楚了就主动结束 — "这个决策已经想清楚了，建议直接进入实现"
- 不触发 writing-plans、不创建 spec 文档
- 不创建 SUMM-Todo 任务

### 适用场景

- "我应该用 Redis 还是 Memcached？"
- "这个 API 应该是 REST 还是 GraphQL？"
- "先做 A 还是先做 B？"
- 任何需要"想清楚"但不需要"设计文档"的时刻

### 不适用

- 需要完整设计方案的新功能（用 brainstorming）
- 已经有明确方案的实现（直接用 to-do-it 或 writing-plans）
- 需要团队共识的决策（grill-me 是个人思考工具）
